# Инструкция для внешнего ассистента: убрать «ошибка сервера» и открыть сайт

Приложение — **Next.js на Vercel**, данные — **PostgreSQL на Neon**. Код при старте **обязательно** читает переменную **`DATABASE_URL`**. Если её нет, строка неверная или таблиц в базе нет, при обращении к страницам/ API часто видна **общая ошибка сервера** (500), без деталей в браузере.

Ниже — что сделать **только во внешних сервисах** (Neon, Vercel), без правок кода.

---

## 1. Что проверить первым делом (Vercel)

1. Зайти на [vercel.com](https://vercel.com) → проект сайта → **Settings** → **Environment Variables**.
2. Убедиться, что для среды **Production** (и при необходимости **Preview**) задано:
   - **`DATABASE_URL`** — полная строка подключения к Postgres (см. шаг 2).
3. После любого изменения переменных сделать **новый деплой**:  
   **Deployments** → три точки у последнего деплоя → **Redeploy** (или пустой коммит в подключённый репозиторий).

Без **`DATABASE_URL`** приложение при импорте модуля БД падает с ошибкой — пользователь видит «ошибка сервера».

**Важно:** в **`DATABASE_URL` не должно быть** `localhost` или `127.0.0.1` — на Vercel это не работает. Нужен хост из Neon.

---

## 2. Строка подключения к базе (Neon)

1. Зайти на [console.neon.tech](https://console.neon.tech) → свой проект → нужная ветка (часто `main`).
2. Открыть **Connect** / **Connection details**.
3. Выбрать режим **Pooled** (пул соединений), формат **URI**.
4. Скопировать строку вида `postgresql://...@...neon.tech/neondb?sslmode=require` (или с параметрами пула).
5. Вставить её в Vercel как значение **`DATABASE_URL`** (без лишних пробелов и кавычек).

---

## 3. Схема базы (Neon SQL Editor)

Если база **пустая** или таблиц нет, любой запрос к данным даст ошибку на сервере.

1. В Neon открыть **SQL Editor**.
2. Открыть в репозитории проекта файл **`scripts/neon-paste-all.sql`** (весь текст файла — это SQL, не имя файла).
3. Скопировать **всё содержимое** и один раз выполнить **Run** в SQL Editor.

После успешного выполнения должны существовать таблицы: `users`, `sessions`, `wallets`, `categories`, и остальные из скрипта.

Если скрипт уже запускали ранее и появились ошибки про «уже существует», ориентироваться на текст ошибки; для **чистой** новой базы достаточно одного полного прогона `neon-paste-all.sql`.

---

## 4. Проверка после деплоя

В браузере открыть:

`https://<ваш-домен-vercel>/api/health`

- **`"ok": true`** и **`"database": "connected"`** — подключение к БД с сервера Vercel работает.
- **`"ok": false`**, **`"database": "missing_env"`** — на Vercel не задан или пустой **`DATABASE_URL`** (вернуться к шагу 1–2 и Redeploy).
- **`"ok": false`**, **`"database": "error"`** — строка подключения неверна, сеть, или схема/права; смотреть логи (шаг 6).

Главная страница и `/admin` дополнительно проверяют сессию и БД; без рабочей Бды они тоже могут отдавать 500.

---

## 5. Вход администратора (если логин не проходит)

В production стартовый пароль админа в БД может быть заглушкой. В коде предусмотрен защищённый вызов:

1. В Vercel задать **`SETUP_SECRET`** (длинная случайная строка).
2. Выполнить **один раз** HTTP-запрос (curl, Postman, и т.п.):

   - Метод: **POST**  
   - URL: `https://<ваш-домен>/api/setup`  
   - Заголовок: `Authorization: Bearer <то_же_значение_что_SETUP_SECRET>`

После успешного ответа пользователь с **`id = 1`** (логин **`slk`**) получает пароль **`31337`** (как в текущей логике setup). Пароль и учётку в интересах безопасности затем сменить через нормальный сценарий приложения, если он появится.

Если **`SETUP_SECRET`** в production не задан, маршрут setup намеренно ведёт себя как несуществующий (**404**) — это норма для безопасности.

Дополнительно рекомендуется задать **`PASSWORD_SALT`** в Vercel (своя длинная строка) **до** появления реальных пользователей; иначе используется соль по умолчанию из кода (см. `.env.example`).

---

## 6. Где смотреть точную причину ошибки

На Vercel: **Project → Deployments → последний деплой → Logs** (или **Functions** / **Runtime Logs**).

Типичные сообщения в логах:

- `DATABASE_URL не задан` — нет переменной в Vercel.
- `[db] DATABASE_URL указывает на localhost` — в проде указан локальный URL.
- Ошибки SQL (`relation "users" does not exist` и т.п.) — не выполнен или не полностью выполнен **`neon-paste-all.sql`**.

---

## 7. Краткий чеклист

| Шаг | Действие |
|-----|----------|
| 1 | Vercel: **`DATABASE_URL`** = Pooled URI из Neon, Redeploy |
| 2 | Neon SQL Editor: один раз весь **`scripts/neon-paste-all.sql`** |
| 3 | Проверка: **`/api/health`** → `ok: true` |
| 4 | При необходимости: **`SETUP_SECRET`**, затем **POST `/api/setup`** с Bearer |
| 5 | Опционально: **`PASSWORD_SALT`**, **`ALLOW_PUBLIC_REGISTRATION=true`** (если нужна публичная регистрация партнёров) |

После этого «ошибка сервера» при открытии сайта обычно пропадает, если проблема была в конфигурации БД или отсутствии схемы, а не в отдельном баге фронта.
